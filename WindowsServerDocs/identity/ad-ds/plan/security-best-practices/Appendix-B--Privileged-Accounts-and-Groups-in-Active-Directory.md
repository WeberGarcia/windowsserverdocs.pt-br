# <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Apêndice b: privilegiadas contas e grupos no Active Directory

>Aplica-se a: Windows Server 2016, Windows Server 2012 R2, Windows Server 2012


## <a name="appendix-b-privileged-accounts-and-groups-in-active-directory"></a>Apêndice b: privilegiadas contas e grupos no Active Directory  
"Privilegiadas" contas e grupos no Active Directory são aqueles aos quais permissões, privilégios e potentes direitos são concedidas que permitam executar praticamente qualquer ação no Active Directory e em sistemas ingressados em domínio. Este apêndice começa discutindo direitos, privilégios, e permissões, seguido por informações sobre o "mais alto privilégio" contas e grupos no Active Directory, ou seja, as contas mais sofisticadas e grupos.  

Também são fornecidas informações sobre contas interno e padrão e grupos no Active Directory, além de seus direitos. Embora recomendações de configuração específica para proteger as contas de privilégio mais alto e os grupos são fornecidas como apêndices separados, este apêndice fornece informações de plano de fundo que ajuda a identificar os usuários e grupos, que você deve se concentrar na proteção. Você deve fazer isso porque eles podem ser aproveitados por invasores comprometer e até mesmo destruir a instalação do Active Directory.  

### <a name="rights-privileges-and-permissions-in-active-directory"></a>Permissões no Active Directory, direitos e privilégios  
As diferenças entre os direitos, permissões e privilégios podem ser confuso e contraditória, até mesmo na documentação da Microsoft. Esta seção descreve algumas das características de cada conforme elas são usadas neste documento. Essas descrições não devem ser consideradas autoritativas para outras documentações Microsoft, porque ele pode usar estes termos de forma diferente.  

#### <a name="rights-and-privileges"></a>Direitos e privilégios  
Direitos e privilégios efetivamente são os mesmos recursos de todo o sistema são concedidos a entidades de segurança, como os usuários, serviços, computadores ou grupos. Em interfaces usadas normalmente por profissionais de TI, geralmente conhecidos como "direitos" ou "direitos de usuário" e, muitas vezes elas são atribuídas pelos objetos de política de grupo. Captura de tela a seguir mostra alguns dos direitos do usuário mais comuns que podem ser atribuídos a entidades de segurança (ele representa o GPO de controladores de domínio padrão em um domínio do Windows Server 2012). Algumas esses direitos se aplicam ao Active Directory, como o **habilitar computador e contas de usuário para serem confiáveis para delegação** direito de usuário, enquanto outros direitos se aplicam ao sistema operacional Windows, como **alterar a hora do sistema**.

#### <a name="permissions"></a>Permissões  
As permissões são controles de acesso que são aplicados a objetos protegíveis como o sistema de arquivos, do registro, serviços e objetos do Active Directory. Cada objeto protegível tem uma lista de controle de acesso associado (ACL), que contém entradas de controle de acesso (ACEs) que concedem ou negar a capacidade de realizar várias operações no objeto de entidades de segurança (usuários, serviços, computadores ou grupos). Por exemplo, as ACLs para muitos objetos no Active Directory contêm ACEs que permitem que os usuários autenticados ler informações gerais sobre os objetos, mas não conceder a capacidade de ler as informações confidenciais ou alterar os objetos.
Com exceção de conta de convidado interna de cada domínio, cada entidade de segurança que faz logon e é autenticada por um controlador de domínio em uma floresta do Active Directory ou uma floresta confiável tem o autenticados usuários segurança SID (identificador) adicionados ao seu token de acesso por padrão. Portanto, se um usuário, serviço ou tentativas de conta de computador para ler as propriedades gerais em objetos de usuário em um domínio, a operação de leitura é bem-sucedida.  

Se uma segurança principal tenta acessar um objeto para que nenhuma ACEs é definidos e que contêm um SID que está presente no token de acesso do objeto, a entidade de segurança não é possível acessar o objeto. Além disso, se uma ACE na ACL de um objeto contiver uma entrada de negação para um SID que corresponda ao token de acesso do usuário, o "Negar" ACE geralmente um conflito de substituição "permitirá" ACE. Para saber mais sobre o controle de acesso no Windows, consulte [o controle de acesso](https://msdn.microsoft.com/library/aa374860(v=VS.85).aspx) no site do MSDN.  

Neste documento, permissões refere-se às funcionalidades quais são concedidas ou negadas para entidades de segurança nos objetos protegíveis. Sempre que houver um conflito entre um direito de usuário e uma permissão, o direito de usuário geralmente tem precedência. Por exemplo, se um objeto no Active Directory tiver sido configurado com uma ACL que impede que os administradores todos acesso de leitura e gravação para um objeto, um usuário que é um membro do grupo de administradores do domínio poderá ver a quantidade de informações sobre o objeto. No entanto, porque o grupo Administradores recebe o usuário certo "apropriar-se de arquivos ou outros objetos", o usuário pode simplesmente assumir a propriedade do objeto em questão e reescreva ACL do objeto para conceder controle total administradores do objeto.  

É por esse motivo que este documento recomenda que você evite usando contas potentes e grupos para administração diária, em vez de tentar restringir os recursos dos grupos e contas. Não é possível efetivamente parar um determinado usuário que tenha acesso às credenciais poderosas que usa essas credenciais para acessar qualquer recurso protegível.  

### <a name="built-in-privileged-accounts-and-groups"></a>Interno privilegiados contas e grupos  
No Active Directory destina-se para facilitar a delegação de administração e o princípio de privilégios mínimos na atribuição de direitos e permissões. "Normais" usuários que possuem contas em um domínio do Active Directory são, por padrão, capaz de ler grande parte o que é armazenado no diretório, mas são capazes de alterar apenas um conjunto muito limitado de dados no diretório. Os usuários que exigem privilégios adicionais podem ser concedidos a associação ao grupo em vários grupos privilegiados que estão integrados ao diretório para que eles podem realizar tarefas específicas relacionadas às suas funções, mas não é possível executar tarefas que não são relevantes para suas obrigações.  

Dentro do Active Directory, há três grupos internos que compõem os grupos de privilégio mais altos na pasta: o grupo Administradores de empresa (EA), do grupo Administradores de domínio (DA) e o grupo de administradores (BA) interno.  

Um quarto de grupo, o grupo Administradores de esquema (SA), tem privilégios que, se abusos, podem danificar ou destruir uma floresta do Active Directory inteira, mas esse grupo é que mais restrito em seus recursos que o EA, DA e BA grupos.  

Além desses quatro grupos, há inúmeras interno adicional e contas e grupos padrão no Active Directory, cada uma delas é concedida direitos e permissões que permitem tarefas administrativas específicas a serem executadas. Embora este apêndice não fornecem uma discussão completa de cada grupo interno ou padrão no Active Directory, ela fornece uma tabela dos grupos e contas que você está mais provável ver suas instalações.  

Por exemplo, se você instalar o Microsoft Exchange Server em uma floresta do Active Directory, grupos e contas adicionais podem ser criados nos contêineres interno e os usuários em seus domínios. Este apêndice descreve apenas os grupos e contas que são criadas nos contêineres interno e os usuários no Active Directory, com base em nativos roles and features. Contas e grupos que são criados pela instalação do software para empresas não estão incluídos.  

#### <a name="enterprise-admins"></a>Administradores corporativos  
O grupo Administradores de empresa (EA) está localizado no domínio raiz da floresta e, por padrão, ele é um membro do grupo Administradores interno em cada domínio na floresta. A conta de administrador interno no domínio raiz da floresta é o único membro padrão do grupo EA. EAs são concedidos direitos e permissões que permitam que afetam as alterações de toda a floresta. Essas são as alterações que afetam todos os domínios na floresta, por exemplo, adicionando ou removendo domínios, estabelecer relações de confiança de floresta ou níveis funcionais de floresta de acionamento. Em um modelo de delegação projetado e implementado corretamente, a associação ao EA é necessária somente quando o primeiro construção da floresta ou ao fazer determinadas alterações de toda a floresta como estabelecer uma relação de confiança de floresta de saída.  

O grupo EA está localizado por padrão no contêiner usuários no domínio raiz da floresta, e é um grupo de segurança universal, a menos que o domínio raiz da floresta está em execução no modo misto do Windows 2000 Server, em cujo caso o grupo é um grupo de segurança global. Embora alguns direitos são concedidos diretamente ao grupo EA, muitos dos direitos desse grupo na verdade são herdados por grupo EA porque ele é um membro do grupo Administradores em cada domínio na floresta. Administradores corporativos não têm direitos nenhum padrão em estações de trabalho ou servidores membros.  

#### <a name="domain-admins"></a>Administradores de domínio  
Cada domínio em uma floresta tem seu próprio grupo Administradores de domínio (DA), que é um membro do grupo de administradores (BA) interno do domínio além de um membro do grupo local Administradores em cada computador que tenha ingressado no domínio. O único membro padrão do grupo para um domínio é a conta de administrador interno para esse domínio.  

DAs são poderosa dentro de seus domínios, enquanto EAs têm o privilégio de toda a floresta. Em um modelo de delegação projetado e implementado corretamente, DA associação deve ser necessária apenas em cenários "vidro quebrar", que são situações em que é necessária uma conta com altos níveis de privilégio em todos os computadores no domínio ou quando determinadas alterações de largura de domínio devem ser feitas. Embora nativos mecanismos de delegação do Active Directory permitem delegação na medida em que é possível usar contas DA somente em situações de emergências, construir um modelo de delegação efetiva pode ser demorado e, muitas organizações usam aplicativos de terceiros para agilizar o processo.  

O grupo DA é um grupo de segurança global localizado no contêiner de usuários para o domínio. Há um grupo para cada domínio na floresta e o único membro de padrão de um grupo DA é a conta de administrador interno do domínio. Como grupo do domínio é aninhado no grupo de BA do domínio e do grupo Administradores local de cada domínio do sistema, DAs não só têm permissões que são especificamente concedidas a administradores de domínio, mas eles também herdam todos os direitos e permissões concedidas ao grupo de administradores do domínio e o grupo local Administradores em todos os sistemas que tenha ingressado no domínio.  

#### <a name="administrators"></a>Administradores  
O grupo de administradores (BA) interno é um grupo de domínio local no contêiner de interno do domínio no qual DAs e EAs são aninhados, e é esse grupo que recebe muitos dos direitos diretos e permissões no diretório e em controladores de domínio. No entanto, o grupo de administradores para um domínio não tem qualquer privilégios em servidores membro ou em estações de trabalho. A associação ao grupo Administradores local de computadores ingressados em domínio é onde privilégio local é concedido; e dos grupos discutidos, somente DAs são membros do grupo Administradores local dos todos os computadores de domínio associado por padrão.  

O grupo Administradores é um grupo de domínio local no contêiner interno do domínio. Por padrão, o grupo de BA do cada domínio contém grupo do EA do domínio raiz da floresta, grupo do domínio local e conta de administrador interno do domínio local. Muitos direitos de usuário no Active Directory e controladores de domínio são concedidos especificamente ao grupo Administradores, não a EAs ou DAs. Grupo de BA do domínio é concedido permissões de controle total sobre a maioria dos objetos de diretório e pode assumir propriedade de objetos de diretório. Embora EA e DA grupos são concedidos determinadas permissões específicas do objeto na floresta e domínios, grande parte o poder dos grupos é realmente "herdado" de sua participação em grupos de BA.  

> [!NOTE]  
> Embora essas são as configurações padrão desses grupos privilegiado, um membro de qualquer um dos três grupos pode manipular o diretório para obter a participação em qualquer um dos outros grupos. Em alguns casos, é trivial para alcançar, enquanto em outros é mais difícil, mas da perspectiva do privilégio potencial, todos os três grupos considerados efetivamente equivalentes.  

#### <a name="schema-admins"></a>Administradores de esquema  
O grupo Administradores de esquema (SA) é um grupo universal no domínio raiz da floresta e tem apenas conta de administrador interno do domínio como um membro de padrão, semelhante ao grupo EA. Embora a associação ao grupo SA pode permitir que um invasor comprometa o esquema do Active Directory, que é a estrutura para toda a floresta do Active Directory, SAs têm alguns direitos de padrão e permissões além do esquema.  

Você deve gerenciar e monitorar a associação ao grupo SA com cuidado, mas em alguns aspectos, esse grupo é "com menos privilégios" que os três mais altos privilegiados grupos descritos anteriormente, pois o escopo de seus privilégios é muito estreito; ou seja, SAs não possuem direitos administrativos em qualquer lugar que não seja o esquema.  

#### <a name="additional-built-in-and-default-groups-in-active-directory"></a>Interno adicional e grupos padrão no Active Directory  
Para facilitar a delegação de administração no diretório, Active Directory vem com vários grupos de interno e o padrão que foram concedidos permissões e direitos específicos. Esses grupos brevemente descritos na tabela a seguir.  

A tabela a seguir lista os grupos internos e padrão no Active Directory. Existem dois conjuntos de grupos por padrão. No entanto, grupos internos estão localizados (por padrão) no contêiner interno no Active Directory, enquanto os grupos padrão estão localizados (por padrão) no contêiner usuários no Active Directory. Grupos no contêiner interno são todos os grupos de domínio Local, enquanto os grupos no contêiner dos usuários são uma mistura de grupos de domínio Local, Global e Universal, além de três contas de usuário individual (administrador, convidado e Krbtgt).  

Além dos grupos privilegiados mais altos descritos anteriormente neste apêndice, algumas contas interno e padrão e os grupos são concedidos privilégios elevados e deve também estar protegido e usado somente em hosts administrativos seguros. Essas contas e grupos podem ser encontradas nas linhas sombreadas na tabela B-1: interno e grupos padrão e contas no Active Directory. Como alguns desses grupos e contas são concedidos direitos e permissões que podem ser usado incorretamente para comprometer o Active Directory ou controladores de domínio, eles têm proteções adicionais conforme descrito em [apêndice c: protegido contas e grupos no Active Directory](../../../ad-ds/plan/security-best-practices/Appendix-C--Protected-Accounts-and-Groups-in-Active-Directory.md).
